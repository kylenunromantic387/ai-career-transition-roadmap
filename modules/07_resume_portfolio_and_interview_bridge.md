# Module 7: Resume, Portfolio, And Interview Bridge

## The Transition Problem

A lot of people learn enough AI to build something, but they still present themselves like beginners.

That is usually a packaging failure, not only a skill failure.

## What Hiring Managers Need To See

They need evidence that you can:

1. solve a real problem with AI
2. make reasonable technical choices
3. evaluate whether the system works
4. understand trade-offs and failure modes
5. translate your previous experience into AI-relevant value

## Resume Reframing

### Weak Positioning

- interested in AI
- completed online AI courses
- passionate about machine learning

### Stronger Positioning

- built and evaluated an LLM-powered internal knowledge assistant for policy search
- implemented retrieval and prompt constraints to reduce unsupported answers
- designed evaluation cases for groundedness and output correctness
- deployed a production-style prototype with logs, feedback, and usage metrics

## How To Convert Past Experience Into AI Language

| Previous Work | AI-Relevant Reframing |
|---------------|-----------------------|
| built backend services | built production systems and APIs for model-backed workflows |
| created dashboards and analyses | built data-driven decision systems and evaluation pipelines |
| wrote test automation | built quality gates, benchmarks, and reliability workflows for AI systems |
| managed product requirements | scoped AI use cases, success criteria, and rollout metrics |
| handled customer operations | understands real workflow pain points for AI automation |

## Interview Story Framework

For every AI project, be ready to answer in this order:

1. What problem were you solving?
2. Why was AI the right tool for that problem?
3. What system did you build?
4. How did you evaluate whether it worked?
5. What failed or behaved unreliably?
6. What trade-offs did you make?
7. What would you improve next?

## Common Interview Gaps

You are underprepared if you cannot answer:

1. Why did you choose that model or approach?
2. How did you know it was working?
3. What were the failure modes?
4. How did you reduce hallucinations or bad outputs?
5. What happens when the data or context is incomplete?

## Practical Rule

Do not present projects as demos.

Present them as decisions, constraints, trade-offs, and outcomes.
